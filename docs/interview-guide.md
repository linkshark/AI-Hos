# AiHos 面试讲解提纲

## 1. 一句话介绍

AiHos 是一个面向医院场景的企业级医疗 RAG + Agent 工作台，解决的是“医院知识如何安全上线、如何被模型稳定引用、如何追踪一次回答为什么慢或为什么命中某份文档”的问题。

## 2. 可以重点讲的项目亮点

### 认证、权限和会话隔离

- JWT 登录注册和刷新机制，接口默认拒绝，按角色开放。
- `memoryId` 由服务端创建并绑定用户，后续聊天、历史恢复、重命名、删除都做归属校验。
- 管理员和普通用户权限隔离，普通用户看不到知识运营和链路诊断。
- 管理员自助注册默认关闭，需要后端演示开关和邀请码同时满足，避免任何人随便注册管理员。

面试表达重点：不是让前端随便传一个会话 ID，而是后端有 owner 表和校验服务，避免跨用户读取或写入历史。

### 混合 RAG 检索

- 关键词召回解决医生名、科室名、年份、版本、指南标题这类精确匹配。
- 向量召回解决口语化表达和语义相近问题。
- 最终通过统一排序合并，生成引用卡片和审计摘要。
- 在线档最终取 6，本地档最终取 4，保证上下文大小和首字等待可控。

面试表达重点：不是只接一个向量库，而是把关键词、向量、metadata、状态过滤和分数拆解组合起来，提高可解释性。

### 知识库发布流

- 文件上传后进入处理态，解析、切片、embedding 完成后进入待发布。
- 管理员发布后才进入线上检索，归档后退出检索。
- metadata 保存和正文编辑分离：只改检索字段不会重新 embedding，正文变更才重建。

面试表达重点：知识库不是“上传即生效”，而是有上线/下线和索引一致性约束，便于运营和回滚。

### 科室 metadata

- 文档科室统一存 `deptCode`，通用文档存 `GENERAL`。
- 页面通过科室树选择，显示中文科室名。
- 父科室覆盖子科室的逻辑已预留，后续可接用户科室授权过滤。

面试表达重点：这不是简单文本字段，而是在为后续院内权限和科室知识域隔离做数据口径统一。

### 链路观测和审计

- 聊天审计记录 provider、memoryId、检索命中、引用数量、首字等待、服务端耗时、TraceId。
- 管理员可在聊天页或后台看链路，普通用户不展示。
- TraceId 可跳 SkyWalking 继续查模型首字等待、检索耗时和接口耗时。

面试表达重点：性能问题不是靠猜，而是有分段 timing 和 trace，能定位慢在检索、模型首字还是流式输出。

### 动态 MCP 接入

- 管理员在后台配置 Streamable HTTP MCP 服务，并测试 `initialize / tools/list`。
- 聊天 Agent 通过 `DynamicMcpTools` 运行时查询已启用工具，再执行 `tools/call`。
- MCP 工具调用会落 `MCP_TOOL_CALL` 审计日志，普通用户不能配置 MCP 服务。

面试表达重点：这不是写死一个天气接口，而是做了管理员配置、运行时发现、工具调用和审计留痕的动态扩展链路。

### 前端体验

- 登录后默认回智能问答页。
- 附件支持点击和拖拽上传，限制 3 个、单个 1MB、总大小 2MB。
- 大图尝试前端压缩；无法压缩或文件超限有明确提示。
- 历史对话支持恢复、置顶、重命名、删除确认和 Markdown 导出。
- 知识库管理和后台运营分开，避免运营诊断工具挤占文件管理主流程。

## 3. 推荐演示路径

1. 登录系统，说明普通用户默认进入智能问答页。
2. 新建会话，切换本地/在线快/在线深模型，说明双模型策略。
3. 提一个院内知识问题，展示回答引用。
4. 上传一个小附件或拖拽文件，说明附件限制和错误提示。
5. 恢复历史对话，展示会话 ID 归属和历史导出。
6. 管理员进入知识库，展示上传、科室选择、metadata 保存、发布上线和归档。
7. 进入知识运营，输入同一个 query，看关键词、向量、最终排序和分数拆解。
8. 进入 MCP 接入，展示天气 MCP 服务测试和工具清单。
9. 回到聊天页询问“杭州明天天气怎么样”，展示动态 MCP 工具调用。
10. 进入审计日志，展示聊天摘要、首字等待、TraceId 和 MCP 工具调用记录。

## 4. 常见追问与回答口径

### 为什么不用纯向量检索？

医院知识里大量问题是精确实体匹配，例如医生名、科室、年份、指南版本。纯向量容易把语义相近但实体不准确的文档排上来，所以这里做了关键词 + 向量混合召回，并在诊断页展示分数来源。

### 为什么 metadata 修改不重新 embedding？

正文没变时向量本身不需要重算。metadata 影响的是过滤、展示和排序，直接刷新文件状态表、chunk 表和内存索引 metadata 即可，避免管理员保存字段后等待长时间重建。

### 为什么在线和本地最终条数不同？

在线模型上下文和质量更强，最终取 6 条引用；本地模型为了减少上下文压力和首字等待，最终取 4 条。召回规则一致，只控制最终上下文大小。

### 为什么要做知识发布流？

医院知识错误上线风险高。上传后先处理和待发布，管理员确认后才参与问答；归档可以快速下线，避免用户继续引用过期内容。

### 当前项目还没做什么？

还没有接真实 HIS/EMR，没有做正式医疗处方闭环，没有把用户科室授权接入真实检索过滤，也没有把检索权重做成后台可编辑配置。这些都可以作为后续路线，而不是当前面试演示版的范围。

## 5. 部署排障速查

- 线上访问空白或 JS chunk 初始化错误：优先确认前端是否完整重新构建并替换旧 `dist`，Nginx 是否还缓存旧 chunk，部署后刷新任意路由是否都回落到 `index.html`。
- MCP 服务 503 或测试失败：确认 MCP URL 是否支持 `POST` JSON-RPC、`initialize -> notifications/initialized -> tools/list` 是否可走通，远端服务如 Render 是否冷启动或限流。
- Ollama 本地回答慢：检查当前模型是否为 `qwen3.5:27b`、`num-ctx` 是否过大、`firstTokenLatencyMs` 是否主要耗在模型首字等待，必要时降到 `num-ctx=4096`。
- Chroma 不通或无召回：确认 Chroma 地址、collection、embedding 模型和知识切片状态；先用知识运营诊断看 keyword/vector/final 三段是否都有结果。
- SkyWalking TraceId 查不到：确认 Java Agent 是否挂载、服务名和实例名是否正确、OAP 地址是否连通，以及当前请求是否经过 `RequestTraceFilter`。

## 6. 最后一句总结

这个项目的重点不是“我调了一个大模型接口”，而是把医疗问答需要的权限、知识发布、混合检索、引用追踪、链路观测和后台运营串成了一个可解释、可维护的系统。
